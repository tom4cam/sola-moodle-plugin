# SOLA v3.9.5 — Release Notes, Feature Summary, and 20-minute Admin Walkthrough

## Part 1: Release Notes (20 April 2026)

v3.9.5 brings full xAI Grok voice support, a multi-provider voice registry so institutions can store and switch between several voice APIs from one screen, a non-destructive "Clear screen" button, a resizable drawer with sane minimums, and an admin-managed custom avatar library.

### Voice providers registry (multi-row)

A new admin setting lets you define one or more voice API providers in a structured table. Each row carries a provider id (`openai` or `xai`), an API key, a friendly label, a default Realtime voice, and a default TTS voice. Three separate active-provider selects pick which label drives Realtime, Text-to-Speech, and Speech-to-Text independently. The legacy single-key configuration keeps working when no rows are defined, so upgrades are backwards compatible.

### Grok Realtime Voice Agent

When the active Realtime provider is xAI, the widget opens a WebSocket to `wss://api.x.ai/v1/realtime` using the stored bearer key directly. When the active provider is OpenAI, SOLA continues to mint an ephemeral `client_secret` via `/v1/realtime/client_secrets` so the key never reaches the browser. Grok voices (Eve, Ara, Leo, Rex, Sal) are available alongside the OpenAI voices in the legacy dropdown.

### Grok TTS

`tts.php` routes to `https://api.x.ai/v1/tts` with the `{text, voice_id, language}` body schema when xAI is the active TTS provider. OpenAI TTS continues to use `/v1/audio/speech` unchanged.

### Grok STT

`transcribe.php` routes to `https://api.x.ai/v1/stt` for student audio when xAI is the active STT provider. OpenAI Whisper is used otherwise.

### Clear screen (replaces Clear history)

The eraser button now wipes only the visible drawer. The full DB transcript is preserved so analytics, AI Analysis Chat, and student learning profiles keep complete conversation records. The label and confirmation prompt were translated across 45 non-English lang files.

### Adjustable drawer with sane minimums

The resize handles now enforce a 320 × 400 minimum so the message list, composer, and header remain readable at any size. The maximum was raised from 90 to 95 percent of viewport to allow wider working areas.

### Custom avatars

A new `customavatars` filearea setting lets admins upload up to 20 PNG or SVG avatars. Uploaded files appear in the avatar dropdown with a "Custom:" prefix and can be set as the site default. Removing a file from the filearea removes it from the dropdown automatically. A new `local_ai_course_assistant_pluginfile()` callback serves the images at the system context.

### Testing

- Jailbreak suite 25 of 25 pass.
- PHP lint clean on every modified file.
- Deployed and verified on dev.sylr.org, dev405.sylr.org, dev500.sylr.org (Moodle 5.0), and dev501.sylr.org (Moodle 5.1 with the `public/` layout).

Plugin version `2026042004`, release `3.9.5`.

## Part 2: Key SOLA Features

### Chat and AI backbone

SOLA (Saylor Online Learning Assistant) is a Moodle local plugin that embeds a floating AI learning coach on course pages. It supports 11 chat providers: Claude, OpenAI, DeepSeek, Gemini, Ollama (local), MiniMax, Mistral, OpenRouter, xAI Grok, Moodle AI (core_ai subsystem), and Custom OpenAI-compatible endpoints. Responses stream via Server-Sent Events. The primary Default AI Provider is configured at the site level and can be overridden per course.

### Comparison providers (LLM picker for admins)

Admins can register additional chat providers in a pipe-delimited table (provider_id, api_key, model list, per-row temperature) so they can compare responses across providers from inside the widget. The picker is admin-only; students never see it.

### Voice

A multi-row voice registry (v3.9.5) stores one or more voice APIs for Realtime, TTS, and STT. Active providers are chosen independently per capability. OpenAI Realtime voices: Shimmer, Alloy, Echo, Fable, Onyx, Nova. Grok voices: Eve, Ara, Leo, Rex, Sal. Whisper and xAI STT handle transcription as a fallback for browsers without Web Speech API.

### Practice quizzes

A full quiz flow includes a setup panel (topic picker or AI-guided), interactive question cards, and a score summary. Practice sessions feed the student learning profile.

### Study plans and reminders

Students can create study plans that schedule email reminders; WhatsApp reminders are available when configured. Inactivity reminders nudge students who have not accessed a course in N days.

### Student learning profiles

After every N student messages in a course, SOLA generates or refreshes a learning profile capturing strengths, weaknesses, learning style, and interests. The profile is injected into the system prompt for personalized coaching.

### Analytics and token cost

The Analytics Dashboard shows cross-course usage, provider comparison, student feedback, and AI Analysis Chat. The Token Cost and Analytics page tracks spend across providers and courses. AI Analysis Scheduled Reports email anonymized insights on a configurable schedule (daily, weekly, monthly).

### Testing Environment (v3.9.3)

An admin page creates a hidden testing course (`SOLATEST`, `visible=0`) and seeds fake students with synthetic AI conversations, ratings, and feedback so admins can preview the Analytics Dashboard and validate plugin changes without affecting real students.

### Safety and moderation

Off-topic detection with configurable warn or end actions, a lockout duration, quiz-hide rules for students and staff, academic integrity nudges with email notifications, wellbeing checks, prompt injection hardening, and an output safety filter that scrubs leaked prompt fragments and API keys. An automated jailbreak test suite runs 25 prompts across 9 attack categories before every release; gates on 25/25 pass.

### Languages and accessibility

46 language packs (en + 45 translations). STT and TTS work in 46 languages with auto-detection and fallbacks. Students can input by voice on supported browsers; the widget supports draggable positioning, a resizable drawer (320 × 400 minimum), and a 2-state expand.

### Branding and UI

Per-site institution name, assistant display name, short name, welcome message, and chat greeting. Position (top/bottom, left/right). Colour picker for avatar colour and fill. Procedurally generated SVG avatars plus 10 built-in image avatars plus admin-uploaded custom avatars (v3.9.5). Per-user avatar preferences are stored alongside the site default.

### Retrieval-augmented generation (RAG)

Semantic search over course content using embedding-based retrieval. Disabled by default. The RAG Admin page shows index status per course with reindex and clear actions.

### Delivery and updates

Settings like CDN URL, CDN version, and GitHub token are configurable. A self-update admin page pulls new releases from the GitHub repo. Analytics export to Redash is available for deeper BI.

### Surveys, rubrics, and usability testing

Optional in-chat student experience surveys, editable via a dedicated admin page. Practice scoring with rubrics for conversation and pronunciation practice. Task-based usability testing for in-widget evaluation.

## Part 3: 20-minute Admin Walkthrough Script (conversational)

Total target time 20 minutes. Each section has a suggested duration and sample voice-over.

### 00:00 to 02:00 — Welcome and framing (2 min)

> "Welcome to SOLA. This is the AI learning coach we have embedded in Moodle. Today I want to give you a tour of the admin surface so you can set up the plugin, configure a default provider, add voice, preview analytics, and create a testing sandbox all in about twenty minutes. We will not touch production data. I'll be showing you the dev site, and everything we do is reversible."

Navigate to **Site administration > Plugins > Local plugins > AI Course Assistant**. Point out the **Settings** category on the left and the external admin pages below it.

### 02:00 to 05:00 — Default chat provider and API key (3 min)

> "The first thing every admin does is pick a Default AI Provider and paste its API key. SOLA supports eleven providers. Most Saylor teams start with OpenAI or Claude."

Open the Settings page. Scroll to **AI Provider and Models**. Choose **Default AI Provider** from the dropdown. Click the password field labelled **API Key**, paste a sandbox key, Save.

> "If your key does not stick on the first click, that's because the click target is the 'Click to enter text' anchor, not the blank space to its left. We tightened that in version 3.9.2 after teachers hit it."

Mention **Model Name**, **API Base URL**, and **Temperature** briefly. Explain that leaving Model Name blank uses the provider default, which stays current as new models ship.

### 05:00 to 08:00 — Comparison providers (LLM picker) (3 min)

> "If you want to compare responses across providers from inside the widget, add rows here. This is admin-only; students never see the picker."

Show the **Comparison providers** table. Add a row with Claude, paste a key, list a couple of models, set a per-row temperature. Save. Reload a course page and open the widget to demonstrate the LLM picker in the composer.

> "Temperature per row (added in v3.9.2) lets you tune creativity per model while comparing. Leave blank to inherit the global default."

### 08:00 to 11:00 — Voice registry and Grok (3 min)

> "Voice is new in 3.9.5. Go to the **Voice providers** section. This is a table where you register one or more voice APIs."

Add a row with **OpenAI**, paste the realtime key, label it `openai-prod`, leave voice fields blank (defaults apply). Add a second row with **xAI**, paste the xAI key, label it `grok-prod`, type `eve` in **Realtime voice** and `eve` in **TTS voice**.

> "Now choose which registered label drives each capability."

Set **Active Realtime provider** to `grok-prod`, **Active TTS** to `grok-prod`, **Active STT** to `openai-prod` (Whisper). Save. Reload a course page with voice mode enabled and demonstrate that the voice tab now talks to Grok for Realtime and TTS, and to Whisper for transcription.

> "You can mix and match. Some institutions pick OpenAI for Realtime and Grok for TTS because of cost or voice preference. The registry is designed to let you switch at any time from one screen."

### 11:00 to 13:00 — Custom avatars and branding (2 min)

> "Let's make it feel like Saylor. Branding lives in the **Branding and UI** section."

Set **Institution Name**, **Assistant Short Name** (SOLA), and the avatar colour. Scroll to **Custom avatars**. Click **Choose a file**, upload a 256 × 256 PNG. Save. Open the **Avatar** dropdown above and point out that the uploaded file now appears with a "Custom:" prefix. Select it as the default. Save.

> "Remove a file from the Custom avatars list to remove it from the dropdown. The built-in ten avatars are always available as a fallback."

### 13:00 to 16:00 — Testing Environment (3 min)

> "Before you turn SOLA on for a live course, seed a hidden testing course so you can see the analytics dashboard with plausible data."

Navigate to **Site admin > Plugins > Local plugins > AI Course Assistant > Testing Environment**. Click **Create hidden testing course**. Notice the green badge saying "hidden". Click Step 2 to seed, leave defaults (15 users, 4 weeks), click **Seed students and chats**. Wait for the success notification.

> "Open the course and open the widget. You will see fake students in the analytics view, spread across the last four weeks, with realistic ratings and feedback. No real learner ever sees this course."

Click **View Analytics for this course**. Walk through the tabs: usage by provider, feedback, AI Analysis Chat.

### 16:00 to 18:00 — Safety, jailbreak test, and AI Analysis (2 min)

> "SOLA ships with prompt injection hardening, an output safety filter that scrubs leaked system prompts and API keys, and an automated jailbreak test that runs 25 prompts across nine attack categories."

Open the CLI if you have terminal access, or simply mention: `php admin/cli/jailbreak_test.php --courseid=2`. Explain it returns 25/25 pass before every release.

Navigate to **Analytics Dashboard**. Open **AI Analysis Chat**. Ask a meta question like "Summarize the top themes from student conversations this week" and show that the answer is grounded in anonymized transcripts. Mention the scheduled report setting for recurring email digests.

### 18:00 to 19:30 — Drawer ergonomics and clear screen (1.5 min)

> "Two small but loved changes: drawer resize and the clear-screen button."

Open the widget. Drag the top-left corner to shrink the drawer; point out that it stops at 320 × 400 px so the composer stays usable. Drag it open to 95 percent of viewport. Click the eraser icon. Read the confirmation: "Clear the visible messages? Your full chat history stays saved." Confirm, and note that the analytics dashboard still contains every message because the DB was not touched.

> "This was an explicit ask: instructors wanted a way to tidy the drawer without losing the transcript that powers learning profiles and analytics."

### 19:30 to 20:00 — Q and A and next steps (0.5 min)

> "That's the tour. Before you go live with SOLA, I recommend three things: one, enable RAG for at least one course and reindex; two, configure AI Analysis Scheduled Reports to land a weekly digest in your inbox; three, add a comparison provider so you can A/B your Default AI Provider against a cheaper or faster alternative over a week of real usage."

Close with:

> "If anything breaks, check Site admin > Notifications and the plugin's GitHub issues at `saylordotorg/moodle-local_ai_course_assistant`. We ship fast and we fix fast."
