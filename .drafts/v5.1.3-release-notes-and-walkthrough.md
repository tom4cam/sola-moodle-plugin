# SOLA v5.1.3

Hotfix release. Voice mode was returning the generic Moodle error "Invalid response value detected" on every attempt to start a realtime conversation; this release fixes the validation type that caused it.

## What's fixed

`classes/external/get_realtime_token.php` declared the `endpoint` field of its return structure as `PARAM_URL`. Both supported providers return `wss://` URLs:

- OpenAI Realtime: `wss://api.openai.com/v1/realtime?model=gpt-realtime-mini`
- xAI via proxy: `wss://<proxy-host>/?token=<JWT>`

Moodle's webservice response-cleaner rejects `wss://` for `PARAM_URL`. The xAI proxy form additionally carries a base64url-encoded JWT in its query string, which compounds the validation failure. The webservice layer raised "Invalid response value detected" before the JS client ever saw the response, so voice mode could not open at all.

Fixed by changing the field type to `PARAM_RAW`. The endpoint is server-generated, not user input, so `PARAM_RAW` is the correct type — there is no untrusted source to sanitise here.

Verified post-fix with `core_external\external_api::clean_returnvalue` against both an OpenAI `wss://` endpoint and an xAI proxy `wss://...?token=<JWT>` endpoint — both validate cleanly.

## Backwards compatibility

Drop-in upgrade from v5.1.2. No setting changes, no schema changes, no behavioural changes outside the voice path. Single-file diff in `classes/external/get_realtime_token.php`.

Plugin version `2026050503`, release `5.1.3`.
